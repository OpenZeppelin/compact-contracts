# compactc front-end blowup on chained Field arithmetic

Repro package for a Compact compiler performance bug that currently blocks
userland SNARK-friendly hashes (Poseidon, MiMC) in Compact source. Prepared
for filing against the compact toolchain; all measurements below are from
this repository's environment.

## Environment

- compact CLI 0.5.1, toolchain 0.31.1 (also reproduced on 0.31.0)
- `compactc.bin` x86_64-unknown-linux-musl, Linux (Fedora 44), 30 GB RAM
- Language version 0.23.0 (`pragma language_version >= 0.23.0`)

## Symptom

The compactc front end (before zkir/keygen: same behavior with `SKIP_ZK=true`)
scales super-linearly — in the worst shape, apparently exponentially — with
the amount of chained `Field` arithmetic inlined into one exported circuit.
Memory grows unboundedly (one 65-round Poseidon instance was killed at
10 min / 5.8 GB RSS). The resulting *circuits* are small (hundreds to a few
thousand rows); only compilation blows up.

## Repros (all in this directory)

Compile each with `compact compile <file> <outdir>`; times measured with
`SKIP_ZK=true` unless noted.

| file | shape | result |
|---|---|---|
| `DiagMdsOnly.compact` | 16 chained 3x3 constant-coefficient linear layers (Poseidon MDS shape) | **>120 s timeout** (8 layers: <1 s) |
| `DiagMdsScalar.compact` | same, struct-free scalars | **>120 s** (structs are not the cause) |
| `DiagMdsOne.compact` | same + multiply-by-asserted-1 per layer | **>120 s** (products distribute over sums before any wire is materialized) |
| `DiagMdsTernary.compact` | same + identity ternary `x == 0 ? 0 : x` per lane | **>120 s** (muxes do not act as sharing barriers either) |
| `DiagMdsWitness.compact` | same + witness hint pinned by asserts per layer | **0.4 s** — witness returns are opaque wires; this is the only working source-level barrier |
| `DiagSbox224.compact` | 224 chained x^5 S-boxes, single lane, no linear layer | 3.1 s (variable*variable chains are fine) |
| `DiagMimcMP.compact` | 110-round MiMC Miyaguchi-Preneel (single lane + per-round key add) | 17.6 s full compile, k=10, rows=598 |
| `MockMimcHang.compact` | real-world case: re-exports `hash4` from `contracts/src/crypto/Mimc` (3 chained compressions, ~330 unrolled rounds) | **>560 s timeout** on 0.31.1 and 0.31.0 |
| `DiagBaseline.compact` | control: `persistentHash` / `transientHash` of 2 and 4 Fields | 5 s; rows below |

Row counts via `zkir mock-compile -v <outdir>/zkir/<circuit>.zkir` (no keygen
needed).

## Why it matters — measured circuit costs

Hash of 2 field elements, identical harness (one ledger write + disclose):

| hash | k | rows |
|---|---|---|
| `persistentHash` (SHA-256, the only upgrade-stable builtin) | 13 | 4,591 |
| userland MiMC x^5 110 rounds (`DiagMimcMP`) | 10 | 598 |
| userland Poseidon, 8 full rounds measured | 9 | 254 |
| userland reference Poseidon via witness-hint workaround (`hash2`) | 12 | 2,233 |
| `transientHash` (platform Poseidon; NOT stable across upgrades) | 7 | 69 |

`persistentHash` of 4 Fields: 7,201 rows; `transientHash` of 4: 113 rows.

Upgrade-stable circuit-efficient hashing in userland is 4-8x cheaper than
SHA-256, but the front-end scaling makes the natural implementations
uncompilable: full Poseidon (65 rounds, 8 full + 57 partial) never finishes,
and MiMC compiles per-compression but hangs once ~330 rounds are inlined
into one entry point — which is what any real consumer circuit (several
hashes per token operation) would do.

## Working workaround (and its cost)

Routing every round's state through a witness hint pinned by asserts
(`DiagMdsWitness` pattern; used by `contracts/src/crypto/Poseidon.compact`)
compiles the full reference Poseidon in ~3 s and matches the official
hadeshash test vector. But it costs ~15-20 rows/round (2,233 vs ~600
achievable) and forces a witness into every consumer's interface, so it is
not a viable library default.

## Suggested fix direction

The data is consistent with expression normalization distributing constant
multiplications over sums (and re-expanding shared subexpressions) without
hash-consing / a canonical linear-combination form. Variable*variable
products appear to act as sharing barriers; constant-linear chains do not.

## Related in-tree code

- `contracts/src/crypto/Mimc.compact` — MiMC hash module (generated by
  `contracts/scripts/generate-mimc.mjs`; pinned golden vectors).
- `contracts/src/crypto/Poseidon.compact` — reference Poseidon with the
  witness-hint workaround (generated by
  `contracts/scripts/generate-poseidon.mjs` from the vendored hadeshash
  instance; verified against the official test vector via
  `contracts/scripts/verify-poseidon.mjs`).
